### M1 — R9.b regression test is mislabeled AND the §11.2 MultiMarkdown lock is missing

- **Where:** `docs/PLAN.md` RTM row R9 (around line 125); `docs/tasks/task-005-05-inline-coerce.md` UC Connection (lines 7–8) and TC-UNIT-06.
- **Finding:** TASK R9 sub-bullets map: **R9.b** = §11.2 MultiMarkdown / PHP-Markdown-Extra → literal text in cells; **R9.c** = §11.3 no `wrap_text=True` on `<br>`. PLAN inverts these (writes "R9.b — `<br>` literal-newline; R9.d — no rich-text Runs"); 005.05 TC-UNIT-06 is tagged `(R9.b lock)` but actually exercises the `<br>`-newline contract which is R9.c. The **real R9.b lock** (MultiMarkdown `[Caption]` rendered as literal text) is **NOT inventoried anywhere in the chain**.
- **Why it matters:** Round-2 reviewer reading the RTM thinks R9.b is closed when it's actually R9.c; the real R9.b gap goes unflagged. A future Developer adding a `[Caption]` parser bypass walks back §11.2 silently.
- **Fix:**
  1. Correct TC-UNIT-06 tag in 005.05: `(R9.c lock — literal \n, not wrap_text=True)`.
  2. Update 005.05 "Use Case Connection" header: change `R9.b` → `R9.c`.
  3. Add new unit test `TC-UNIT-22 (TestHonestScopeLocks::test_multimarkdown_caption_literal)`: input markdown with `[Caption]` text → assert cell value is plain string `"[Caption]"`, no caption-metadata side channel.
  4. Correct PLAN RTM R9 row: split into `(R9.b — MultiMarkdown literal text; R9.c — <br> literal-newline + no wrap_text; R9.d — no rich-text Runs)`.

  Cost: ~5 LOC test + 4 small text edits. Does not change chain shape.
