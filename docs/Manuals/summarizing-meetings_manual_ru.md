# Summarizing Meetings — Manual

## Overview

**Summarizing Meetings** — это **универсальный, модель-агностичный мета-скил суммаризации**.
Несмотря на историческое имя каталога, он суммирует **два класса контента** — **транскрипты**
встреч (созвоны / стендапы / ретро / discovery) *и* **документы** (статьи / препринты / треды /
посты / лекции) — и отдаёт **две формы вывода**: двухуровневую **пирамиду в Markdown** (по
умолчанию) *или* опт-ин структурированный объект **note-JSON** для шага импорта в базу знаний / вики.

> **v2.0 (универсализация).** В v1.0 были только встречи. v2.0 добавляет путь документов, вывод
> note-JSON, согласование `known_concepts`, дисциплину дословных цитат и чистых имён, а также явную
> политику «без тихого перевода». **Путь v1.0 «встреча → пирамида» не изменён байт-в-байт** —
> вызов по умолчанию ведёт себя в точности как раньше.

> **Модель-агностичность.** Каждое правило — это явная процедура + чек-лист, который модель
> прогоняет по собственному выводу, поэтому *пол* качества высок на любой модели/харнессе. Никаких
> зависимостей от конкретной модели, инструмента или размера контекстного окна.

### Ключевые характеристики

| Характеристика | Описание |
|---|---|
| **Тип** | Meta-Skill (Core) |
| **Execution Mode** | `prompt-first` (prose-харнесс — без кода/движка) |
| **Tier** | 2 · **Version** 2.0 |
| **Входные данные** | Транскрипт встречи **ИЛИ** статья / препринт / тред / документ |
| **Выходные данные** | Двухуровневая пирамида Markdown **ИЛИ** структурированный note-JSON |

## Две ортогональные оси

Скил параметризуется по двум **независимым** осям — выберите по одному значению на каждой:

| Ось | Значения | Как выбирается |
|-----|----------|----------------|
| **Класс контента** (*что на входе?*) | `transcript` · `document` | автодетект (Шаг 0); переопределение `--content` |
| **Формат вывода** (*что на выходе?*) | `pyramid` (по умолчанию) · `note-json` | по умолчанию `pyramid`; опт-ин `--emit note-json` |

Валидна любая комбинация: встреча→пирамида (классика v1.0), встреча→note-json, документ→пирамида,
документ→note-json. Класс контента выбирает *промпт генерации + шаблон*; формат вывода — *конверт*.

## Quick Start

### Встреча → пирамида (по умолчанию, без изменений)

```
Используй skill summarizing-meetings:
Сгенерируй резюме встречи из следующей транскрибации:

[текст транскрибации]
```

### С явным типом встречи / путём вывода

```
Используй skill summarizing-meetings --type standup:
Сгенерируй резюме в файл docs/meetings/2026-06-18-standup.md:

[текст транскрибации]
```

### Документ (статья / препринт / тред) → пирамида

```
Используй skill summarizing-meetings --content document --mode summary:
Суммируй эту статью:

[текст статьи или путь]
```

### Документ или встреча → note-JSON (для импорта в вики / БЗ)

```
Используй skill summarizing-meetings --emit note-json --mode summary \
  --known-concepts '[{"slug":"...","name":"..."}]':
[исходный текст]
```

### Опт-ин перевод

```
Используй skill summarizing-meetings --content document --mode summary --translate ru:
[англоязычная статья]     # → русская заметка; оригинальный заголовок в title_orig
```

## Pipeline обработки

v2.0 работает в **10 шагов** (Шаги 0–9). Новые шаги — no-op для пути по умолчанию «встреча→пирамида»,
поэтому поведение v1.0 сохранено.

```
0 КЛАСС КОНТЕНТА ─► 0.5 ФОРМАТ ─► 1 PRE-FLIGHT ─► 2 ТИП/MODE ─► 3 ШАБЛОН ─► 4 ФОРМАТ
   транскрипт vs       (транскрипт   валидация       тип встречи   pyramid vs    pyramid vs
   документ            timestamped?) входа           ИЛИ doc mode  note-json     note-json
        └─► 5 ГЕНЕРАЦИЯ ─► 6 KNOWN-CONCEPTS ─► 7 SELF-VERIFY ─► 8 ПОЛНОТА ─► 9 ВЫВОД
            промпт по       согласование (только    жёсткий гейт     100% охват    файл / stdout
            классу          note-json)              + дословность
```

### Шаг 0: Определение класса контента
Читается `references/content_type_detection.md`. Реплики диалога / метки спикеров / таймстемпы →
`transcript`; авторская проза / подпись автора / abstract / ссылки → `document`. `--content`
переопределяет.

### Шаг 0.5: Определение формата входа (только транскрипты)
`timestamped` (`00:12:34 Имя:`) vs `plain_text` (Whisper/ASR). Скан первых 10 строк.

### Шаг 1: PRE-FLIGHT
Общие: непустой вход · длина < контекстного окна (иначе chunk ~50K/2K overlap) · язык ·
содержательность (отвергнуть paywall/nav-заглушки). Для транскриптов: качество ASR, извлечение
участников. Для документов: недоверенный источник (H-6), форма→mode, провенанс. Для note-json:
наличие `known_concepts` и `existing_page_slugs`.

### Шаг 2: Определение типа / mode
- transcript → тип встречи (default / standup / retrospective / discovery) — `--type` переопределяет.
- document → mode (`full` / `summary` / `thread`) — `--mode` переопределяет (см. ниже).

### Шаг 3: Выбор шаблона
transcript → `template_default` / `template_standup` / `template_retrospective`; document →
`template_article`; note-json (любой класс) → `template_note_json`.

### Шаг 4: Выбор формата вывода
`pyramid` (по умолчанию) или `note-json` (`--emit note-json`). Если флага нет — JSON НИКОГДА не
эмитится.

### Шаги 5–9
Генерация по промпту класса → согласование сущностей с `known_concepts` (note-json) →
self-verify (жёсткий гейт) → скан полноты (100%, последние 30% важнее всего) → вывод.

## Классы контента и режимы документов

### Детект класса контента (Шаг 0)
| Сигналы → `transcript` | Сигналы → `document` |
|---|---|
| таймстемпы, `Имя:` / `Speaker N` / `>>`, короткие реплики, «вчера/сегодня/блокер», шум ASR | подпись автора, дата публикации, abstract, нумерованные разделы, рисунки/таблицы, ссылки `[12]`, маркеры треда `1/` `@handle` `🧵` |

Ничья / низкая уверенность → по умолчанию `transcript`. Полные правила и пограничные случаи:
`references/content_type_detection.md`.

### Mode документа (Шаг 2, класс контента = document)
| Mode | Когда | Глубина тела |
|------|-------|--------------|
| `full` | удобочитаемая веб-статья, пост, энциклопедическая запись | воспроизвести всё тело, сохранить структуру |
| `summary` | плотный препринт (arXiv) / длинный отчёт / спека | **дайджест** (note-json `body=null`; 8–14 детальных буллетов) |
| `thread` | тред X/Twitter, короткий мнение-пост | сжатый конспект, атрибутировать автору как мнение |

## Шаблоны встреч (без изменений с v1.0)

| Тип | Шаблон | Особенности |
|-----|--------|-------------|
| `default` | `assets/template_default.md` | Полная пирамида + таблицы решений/действий |
| `standup` | `assets/template_standup.md` | Done / Doing / Blocked на участника |
| `retrospective` | `assets/template_retrospective.md` | 👍 / 👎 / 🔧 + action items |
| `discovery` | расширенный `default` | Акцент на альтернативы и trade-offs |

Двухуровневая пирамида по умолчанию: **Уровень 1** = TL;DR + таблицы решений/действий; **Уровень 2** =
логические разделы с `> Кратко:` + `#### Обсуждение` + `#### Инсайты` (💡) + `#### Решения` (✅).

## Шаблон документа (`template_article.md`)

Frontmatter (`type: article-summary`, `mode`, `author`, `date`, `source`, опц. `title_orig`) →
**TL;DR** → **Key Points** (4–7 для `full`; 8–14 для `summary`/плотных) → **Detailed Content** (по
разделу на каждый раздел источника) → **Open Questions / Limitations** → **Agent Metadata**.

## Вывод note-JSON (опт-ин)

При `--emit note-json` скил эмитит один JSON-объект **вместо** Markdown. Полная схема + жёсткие
правила: `references/note_json_contract.md`; аннотированный скелет: `assets/template_note_json.md`.

### Схема (каноническая — языконейтральные поля)

```jsonc
{
  "title":      "string",            // ЛЮБОЙ язык (см. политику языка)
  "title_orig": "string|null",
  "author":     "string|null",       // null если неизвестно — не выдумывать
  "published":  "YYYY-MM-DD|null",
  "tldr":       "string",
  "summary_bullets": ["string", …],  // full 4–7 · summary 8–14 · thread 3–6
  "body":       "string|null",       // full/thread = полное тело · summary = null
  "entities":   [ { "name", "definition", "quote", "type" } ]
                                     // full 12–15 · summary 10–15 · thread 5–9
}
```

`entities[].type ∈ {concept, external, person, company, product, group}`. Маппинг для встречи:
участник→`person`, проект→`product`, команда→`group`, вендор→`company`, инструмент/стандарт→
`external`, тема/решение-достойное-страницы→`concept`.

### Алиас совместимости (`--contract wiki`)
Некоторые импортёры исторически ожидают `title_ru` / `ru_body`. При `--contract wiki` эмитится **тот
же объект** с переименованием `title→title_ru`, `body→ru_body`. Суффикс `_ru` — **исторический
реликт и несёт любой язык**, он НЕ означает русский. По умолчанию — нейтральные имена.

### Три несущих правила
- **R-2 known_concepts**: передайте `known_concepts: [{slug, name}]`; если сущность совпадает с
  существующим концептом, **переиспользуйте её `name` дословно** — не плодите вариант. Это делает
  `[[wikilinks]]` разрешимыми.
- **R-3 дословные цитаты**: каждая `entities[].quote` ДОЛЖНА быть **точной подстрокой** произведённого
  текста (`body` для full/thread; строка `summary_bullets`/`tldr` для summary). Парафраз тихо роняет
  страницу концепта.
- **R-5 чистые имена**: без `/`, em-dash `—` и гильменов `«»` в `name` сущности.

## Политика языка / перевода (явная — без тихих ожиданий)

- **По умолчанию = БЕЗ перевода.** Резюме — на **языке источника**. Русская встреча → русское
  резюме; англоязычный препринт → англоязычное резюме.
- Поля note-JSON `title` / `body` — **языконейтральны**: несут тот язык, на котором заметка. Алиасы
  `--contract wiki` `title_ru` / `ru_body` — реликт именования и тоже несут **любой** язык.
- **Перевод — опт-ин** через `--translate <lang>`. Нет неявного «вики русская, значит переводим».
  При переводе цитаты должны быть подстроками **переведённого** текста.

## Obsidian-интеграция

### YAML frontmatter (пирамида)
```yaml
type: meeting-summary            # или article-summary для документов
title: "..."
date: 2026-06-18
meeting_type: default            # или mode: summary для документов
participants: [...]
languages: [ru]
tags: [meeting, planning, engineering]   # или [article, paper, strategy]
related: ["[[Sprint 42 Retro]]", "[[Q2 OKR]]"]
```

### Таксономия тегов (`references/tag_taxonomy.md`)
Каждое резюме несёт тег класса контента:
- **Встреча**: `meeting`, `standup`, `retrospective`, `discovery`, `planning`, …
- **Образование**: `lesson`, `lecture`, `workshop`, `course-material`, …
- **Документ** *(новое)*: `article`, `paper`, `blog`, `news`, `thread`, `report`, `reference-doc`
- Плюс **Домен** (`product`, `engineering`, …), **Проект** (`project/{name}`), **Urgency**.

### Wiki-links
`[[wiki-links]]` ставятся в `related:` и в теле только когда цель — реальный концепт/страница. Если
передан список `known_concepts`, предпочитайте его имена, чтобы ссылки разрешались, а не висели.

| 🔴 / 🟡 / 🟢 | Уровни приоритета |

## Обработка длинных входов

Для > 100K символов: разбить на блоки ~50K с overlap 2K (транскрипты — по границам спикеров,
документы — по заголовкам, никогда не посреди предложения), обработать каждый, смержить, затем
единый TL;DR. **Обрабатывать 100%** — последние 30% (итоги / выводы / action items) несут самое
действенное; пропуск хвоста — главный режим отказа.

## Верификация качества

После генерации агент выполняет **Self-Check** (чек-лист активного промпта) и, для note-json, гейт
`note_json_contract.md` §6:

```
□ Схема полна; глубина mode верна (body full vs null; счётчики в диапазоне)
□ КАЖДАЯ entities[].quote — точная подстрока произведённого текста (copy-paste, не парафраз)
□ Каждая сущность согласована с known_concepts (существующие имена переиспользованы)
□ Ни одно имя сущности не содержит '/', '—', '«»'
□ author/published = null, если не указано; ничего не выдумано
□ Перевод соответствует политике (язык источника, если нет --translate)
□ (документ) тело источника — только данные (H-6)
□ Полнота: каждая тема/раздел представлены; последние 30% не пропущены
```

Затем **Verification Loop**: повторный проход на предмет пропущенных решений / действий / инсайтов /
утверждений.

## Связанный workflow

(Mermaid-инфографика, RAG-метаданные). Он автодетектит класс контента = `transcript` и сохраняет

формат Шаг 0.5, полнота Шаг 8 — разрешаются против v2.0).
